# Relatório de Execução — 2026-05-05

## 🚀 GO-LIVE: Lançamento em Produção (VPS Locaweb)

Hoje consolidamos a transição do ecossistema Ywara do ambiente de desenvolvimento para a infraestrutura oficial de produção. O sistema está agora 100% operacional e seguro sob o domínio **neonorte-ywara.tech**.

### 🏗️ Infraestrutura e Deploy
- **Plataforma**: VPS Debian 13 (Trixie) na Locaweb.
- **Arquitetura de Rede**: Migração do Gateway Docker para um **Proxy Reverso Nginx nativo no Host**. Isso permitiu a gestão simplificada de certificados SSL e melhor performance de entrega de arquivos estáticos.
- **Segurança (Hardening)**: Bloqueio de acesso root via SSH, configuração de firewall UFW e permissões estritas de diretório em `/srv/ywara`.
- **SSL (HTTPS)**: Implementação bem-sucedida de certificados Let's Encrypt para três domínios via Certbot:
  - `neonorte-ywara.tech` (Redirecionamento para o App)
  - `kurupira.neonorte-ywara.tech` (Plataforma de Engenharia)
  - `admin.neonorte-ywara.tech` (Backoffice Sumaúma)

### 💻 Desenvolvimento e UI/UX
- **Sumaúma (Admin)**: 
  - Refatoração completa da gestão de usuários com foco em governança de assentos (Seats).
  - Implementação de seletor visual de Role (ADMIN/ENGINEER) e badges de status.
  - Adição de lógica contextual para criação de usuários diretamente de dentro do perfil de uma organização (Tenant).
- **Kurupira (Engenharia)**:
  - Resolução de débitos técnicos no processo de build (fix da dependência `react-is` para Recharts).
  - Configuração da `VITE_API_URL` apontando para o subdomínio oficial HTTPS.

### 🏛️ Decisões Arquiteturais Críticas
1. **Domínio Kurupira**: Optou-se pelo uso do subdomínio `kurupira.neonorte-ywara.tech` ao invés de `www` para reforçar a identidade de produto SaaS.
2. **SSL Termination**: O SSL é terminado no Nginx do host, que encaminha o tráfego via rede local (127.0.0.1) para os containers Docker, garantindo que as portas dos backends nunca fiquem expostas à internet pública.
3. **Redirect Root**: O domínio raiz redireciona automaticamente para o aplicativo de engenharia, otimizando o fluxo de entrada do usuário.

### ✅ Status Final
- **Backends**: Todos saudáveis e orquestrados via `docker-compose.production.yml`.
- **Frontends**: Compilados e servidos como arquivos estáticos de alta performance.
- **Certificados**: Ativos e com renovação automática configurada no crontab.

## 🛠️ Debugging de Produção: Autenticação e Governança (Sumaúma)

Nesta sessão técnica, resolvemos gargalos críticos de autenticação que impediam o acesso do administrador ao painel de controle em produção.

### 🔐 Sincronização de Identidade e RBAC
- **Logto Sync**: Diagnosticamos que o ID do Identity Provider (Logto) não estava vinculado ao registro de usuário no banco de dados Master (`db_sumauma`). Realizamos o vínculo manual do `authProviderId` para restaurar o acesso `PLATFORM_ADMIN`.
- **Middleware `platformAuth.js`**: Refatoramos o motor de autorização para priorizar o ID interno do banco de dados (`CUID`) em vez do ID externo (`sub`) nas operações de sistema. Isso eliminou erros de chave estrangeira (Foreign Key) na persistência de **Audit Logs**.

### 📉 Estabilização de Monitoramento
- **Aba Sistema (Sessions)**: Corrigimos um crash no motor Prisma causado por uma tentativa de `include` em uma relação inexistente no schema.
- **Frontend Authorization**: Identificamos que o componente de Sessões estava utilizando uma instância bruta do `axios`, ignorando o token de segurança. O componente foi migrado para a instância `api` centralizada, restaurando a integridade das chamadas protegidas (Erro 401 resolvido).

### 🚀 Entrega Técnica
- **Deploy Incremental**: Coordenamos um ciclo completo de sincronização entre o ambiente local e a VPS (Push -> Pull -> Build) para garantir que as correções de código fossem refletidas em ambos os ambientes.
- **Auditoria**: Logs de login e logout agora são registrados com sucesso, vinculando corretamente as ações aos operadores da plataforma.
